# MVP3 Project Framework

이 디렉터리는 "3LLM 협업 워크플로우"를 위한 기본 구조입니다.
모든 문서는 `docs/` 내에 있으며, 각 에이전트의 역할은 `AGENTS.md`에 정의되어 있습니다.

---

## 🚀 시작하기: 일일 업무 워크플로우

모든 에이전트는 매일 작업을 시작할 때 아래 순서에 따라 정보를 동기화하고, 자신의 할 일을 찾아 진행합니다.

### **1단계: `INBOX` 확인**
- **파일:** `docs/INBOX.md`
- **목적:** 자신에게 온 새로운 메시지(`Unread` 섹션)나 `@ALL` 공지가 있는지 가장 먼저 확인합니다. 다른 팀원의 요청이나 피드백에 대응하는 것이 최우선입니다.

### **2단계: `DECISIONS` 확인**
- **파일:** `docs/DECISIONS.md`
- **목적:** 새로운 기능 제안(DT)이 올라왔는지, 또는 기존 제안에 대해 S(PO)님의 최종 결정이 내려졌는지 확인합니다. 승인된 DT는 곧 `WORK.md`의 태스크로 전환됩니다.

### **3단계: `WORK` 확인**
- **파일:** `docs/WORK.md`
- **목적:** 자신에게 할당된 태스크의 현재 상태(`Todo`, `Doing` 등)와 우선순위를 확인합니다. 오늘 진행할 작업을 명확히 합니다.

### **4단계: `STATUS` 업데이트 및 작업 진행**
- **파일:** `docs/STATUS.md`
- **목적:** 작업을 시작하기 전, `STATUS.md`에 본인의 현재 작업 상태를 기록하여 팀원들과 공유합니다. 그 후, `WORK.md`에 기록된 자신의 태스크를 진행합니다.

---

## ✨ 신규 기능/아이디어 제안 워크플로우

새로운 아이디어를 구체적인 작업으로 전환하는 프로세스는 `docs/RULES.md`의 "통합 개발 워크플로우"를 따릅니다.

### **1. 제안 (DT 생성)**
- **장소:** `docs/DECISIONS.md`
- **방법:** 새로운 아이디어를 '의사결정 토큰(DT-XXX)' 형식에 맞춰 작성합니다. '문제'와 '제안'을 명확히 설명해야 합니다.

### **2. 논의 (피드백 요청)**
- **장소:** `docs/INBOX.md`
- **방법:** `@ALL` 채널에 새 DT가 등록되었음을 알리고, 다른 팀원들에게 기술적/정책적 검토를 요청합니다. 모든 논의는 DT의 `Reviews` 섹션에 기록됩니다.

### **3. 결정 (S의 승인)**
- **장소:** `docs/DECISIONS.md`
- **방법:** S(PO)님이 모든 검토 의견을 바탕으로 최종 결정을 내립니다.

### **4. 실행 (Task 생성)**
- **장소:** `docs/WORK.md`
- **방법:** 승인된 DT는 `WORK.md`에 공식 '태스크(TASK-XXX)'로 생성되며, 이때 `Related Decision: DT-XXX` 필드를 통해 추적성을 확보합니다. 이후 담당자는 이 태스크를 기반으로 실제 작업을 진행합니다.
